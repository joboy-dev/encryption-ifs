================================================================================
  BLOCKCHAIN CONNECTION ERROR — DIAGNOSIS & FIX
  Error: "orderer client failed to connect to orderer.example.com:7050:
          failed to create new connection: context deadline exceeded"
================================================================================

ROOT CAUSE
----------
The peer CLI successfully discovers the orderer endpoint from the channel config
(orderer.example.com:7050), but fails to establish a TCP/TLS connection to it.
This is caused by TWO issues working together:

  1. CODE BUG (FIXED): The `record_on_blockchain` invoke command was missing
     the `-o localhost:7050 --tls --cafile <orderer-tls-ca>` flags. Without
     these, the peer CLI tries to connect to the orderer hostname from the
     channel config ("orderer.example.com:7050") instead of "localhost:7050".
     Since the DNS name "orderer.example.com" typically only resolves inside
     the Docker network, it times out from the host machine.

  2. CODE BUG (FIXED): The `get_blockchain_record` function was using the
     wrong binary path (`FABRIC_PATH/bin/peer` instead of
     `FABRIC_SAMPLES_PATH/bin/peer`) and was missing the `env=` argument
     in subprocess.run(), so it wouldn't pick up TLS certs or MSP identity.

WHAT WAS FIXED IN CODE
-----------------------
  - record_on_blockchain: Added `-o localhost:7050 --tls --cafile <orderer-ca>`
    so the peer CLI connects to the orderer via localhost (port-forwarded from
    Docker) rather than trying the unresolvable Docker hostname.

  - get_blockchain_record: Fixed binary path from FABRIC_PATH to
    FABRIC_SAMPLES_PATH, and added `env=cls._get_fabric_env()` so it picks up
    MSP identity, TLS certs, and peer address.

  - Both functions: Cast user_id to str() to prevent type errors if an integer
    ID is passed from the database model.


IF THE ERROR PERSISTS — INFRASTRUCTURE CHECKS
----------------------------------------------
If you still see "context deadline exceeded" after the code fix, the issue is
that the Fabric network itself is not reachable. Run through these checks:

1. VERIFY THE FABRIC NETWORK IS RUNNING
   ─────────────────────────────────────
   cd /home/korede/fabric-samples/test-network
   docker ps --format "table {{.Names}}\t{{.Status}}\t{{.Ports}}"

   You MUST see these containers running:
     - orderer.example.com          (port 7050)
     - peer0.org1.example.com       (port 7051)
     - peer0.org2.example.com       (port 9051)

   If they are not running, bring up the network:
     ./network.sh up createChannel -c mychannel2 -ca

   If the network was torn down, you also need to redeploy chaincode:
     ./network.sh deployCC -ccn basic -ccp ../asset-transfer-basic/chaincode-go -ccl go -c mychannel2


2. VERIFY ORDERER PORT IS ACCESSIBLE ON LOCALHOST
   ───────────────────────────────────────────────
   curl -v telnet://localhost:7050 2>&1 | head -5

   Or:
     nc -zv localhost 7050

   If the port is not open, the orderer container may have stopped or its
   port mapping is wrong. Check:
     docker logs orderer.example.com --tail 50


3. VERIFY PEER PORT IS ACCESSIBLE
   ───────────────────────────────
   nc -zv localhost 7051

   If this fails, the peer container is down.


4. VERIFY THE CHANNEL EXISTS
   ─────────────────────────
   export FABRIC_CFG_PATH=/home/korede/fabric-samples/config/
   /home/korede/fabric-samples/bin/peer channel list

   Confirm "mychannel2" appears in the list. If not, the channel was never
   created or the network was reset.


5. VERIFY CHAINCODE IS DEPLOYED
   ─────────────────────────────
   /home/korede/fabric-samples/bin/peer lifecycle chaincode queryinstalled

   Confirm that "basic" chaincode is installed and committed.


6. CHECK TLS CERTIFICATE FILES EXIST
   ──────────────────────────────────
   ls -la /home/korede/fabric-samples/test-network/organizations/ordererOrganizations/example.com/orderers/orderer.example.com/msp/tlscacerts/tlsca.example.com-cert.pem
   ls -la /home/korede/fabric-samples/test-network/organizations/peerOrganizations/org1.example.com/peers/peer0.org1.example.com/tls/ca.crt
   ls -la /home/korede/fabric-samples/test-network/organizations/peerOrganizations/org1.example.com/users/Admin@org1.example.com/msp/

   If any of these are missing, the crypto material was deleted. You need to
   bring the network down and back up:
     ./network.sh down
     ./network.sh up createChannel -c mychannel2 -ca


7. QUICK SMOKE TEST
   ─────────────────
   Run a manual query from the command line to isolate whether it's a code
   issue or an infrastructure issue:

   export FABRIC_CFG_PATH=/home/korede/fabric-samples/config/
   export CORE_PEER_TLS_ENABLED=true
   export CORE_PEER_LOCALMSPID="Org1MSP"
   export CORE_PEER_MSPCONFIGPATH=/home/korede/fabric-samples/test-network/organizations/peerOrganizations/org1.example.com/users/Admin@org1.example.com/msp
   export CORE_PEER_TLS_ROOTCERT_FILE=/home/korede/fabric-samples/test-network/organizations/peerOrganizations/org1.example.com/peers/peer0.org1.example.com/tls/ca.crt
   export CORE_PEER_ADDRESS=localhost:7051

   /home/korede/fabric-samples/bin/peer chaincode query \
     -C mychannel2 -n basic \
     -c '{"Args":["GetAllAssets"]}'

   If this succeeds → the network is fine and the code fix should resolve it.
   If this fails   → the Fabric network needs to be restarted/redeployed.
